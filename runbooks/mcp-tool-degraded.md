# Runbook: MCP tool error rate spike

## What you're looking at

A spike in `metric=mcp.tool.errors` for one tool — or a Sentry alert reporting elevated `error_code` rates from a specific MCP tool. Could surface as:

- Multiple `mcp.tool.errors{tool=<name>, error_code=<X>}` lines in [Fly logs](https://fly.io/apps/cyggie-gateway/logs).
- A user (in Slack or directly via Claude Desktop) reporting "Cyggie searches always fail now."
- The `mcp_audit` table showing a sudden drop in `ok=true` rate for one `tool_name`.

The MCP server exposes 6 read tools (`cyggie_search`, `cyggie_get_company`, `cyggie_get_contact`, `cyggie_recent_meetings`, `cyggie_get_meeting`, `cyggie_get_notes`) plus the flagged `cyggie_execute_sql`. Per [`api-gateway/src/mcp/server.ts`](../api-gateway/src/mcp/server.ts), each tool can fail with one of a stable enum of error codes (`NOT_FOUND`, `AMBIGUOUS`, `PERMISSION_DENIED`, `INVALID_INPUT`, `TOOL_DISABLED`, `TIMEOUT`, `INTERNAL`).

The error code tells you a lot about what's wrong. Branch on it first.

## Identify the failure mode by error_code

Query the audit log to see what's going wrong:

```sql
SELECT tool_name, error_code, COUNT(*) AS n,
       MIN(created_at) AS first_seen,
       MAX(created_at) AS last_seen
FROM mcp_audit
WHERE created_at > NOW() - INTERVAL '1 hour'
  AND ok = false
GROUP BY tool_name, error_code
ORDER BY n DESC;
```

### `INTERNAL` — uncaught exception

Most concerning. Means the tool handler threw something the envelope bridge didn't catch as a known shape.

1. Open Sentry. Filter `tags:surface:mcp` for the last hour. Find the stack trace.
2. Common causes:
   - Drizzle query throwing because the schema migration didn't run (`column does not exist`).
   - The fuzzy resolver passed a `null` to a function expecting a string.
   - Anthropic SDK timeout on a `cyggie_ask`-like fallback path.
3. Fix the root cause; don't suppress the error. `INTERNAL` is the catch-all — every occurrence is a bug.

### `TIMEOUT` — Postgres `statement_timeout` (5s) tripped

Almost always on `cyggie_execute_sql` (the only tool with an explicit DB timeout). Means a user-generated SQL query took >5s.

1. Pull the offending query from `mcp_audit.input_summary` (the SQL tool writes the full query untruncated).
2. If it's an unbounded scan, the tool description should already discourage it; consider adding a more explicit caveat.
3. If it's hitting a slow query plan, check whether Neon's planner is missing statistics (rare; `ANALYZE` the table).
4. Tool is correctly returning a user-friendly envelope; the user sees "Query exceeded 5s timeout."

### `PERMISSION_DENIED` (Postgres 42501) — query touched a denied table

Two cases:

1. **Legitimate user query** asking about `users` / `oauth_*` / `slack_user_mappings` / etc. Tool description should already list allowed tables; user just got it wrong. Audit row is fine, user sees clear error.
2. **Prompt-injection attempt** — query was generated by an LLM trying to extract secrets. Per plan slice 10, every `PERMISSION_DENIED` Sentry-captures with tag `security: sql_permission_denied`. Worth reviewing patterns over time.

### `INVALID_INPUT` — pre-flight rejected

The query failed `cyggie_execute_sql`'s pre-flight check (contains `;`, doesn't start with `SELECT`/`WITH`, exceeds 8KB). Or some other tool's Zod input validation failed.

Almost never a server bug — usually the LLM generated bad input. If you see a sustained spike from one client, file a bug against that client's prompt.

### `NOT_FOUND` / `AMBIGUOUS`

Expected error categories from the fuzzy resolver. High rates mean the resolver isn't matching what users (or their LLMs) expect.

1. Check the input pattern. Are users asking for company names with typos? Half-remembered names?
2. The resolver in [`api-gateway/src/mcp/resolvers.ts`](../api-gateway/src/mcp/resolvers.ts) uses ID lookup → exact normalized → ILIKE substring. Consider whether the substring strategy is missing common variants.

### `TOOL_DISABLED`

User invoked `cyggie_execute_sql` while `CYGGIE_MCP_SQL_ENABLED=false` (default in production). Expected. Educate the user; don't change the flag without thinking.

## Triage by symptom

### Single tool, single error_code, single user

Likely a user-specific issue (their query, their client). Pull the user's recent audit rows; see if there's a pattern.

### Multiple tools, same error_code (especially `INTERNAL`)

Cross-cutting issue. Recent deploy? Check if a shared service (resolvers.ts, db.ts, envelope-bridge.ts) was touched. Roll back if necessary.

### Single tool, high error_code variance

Suggests an upstream dependency degraded. For data tools, check Neon dashboard for query latency / pool saturation. For `cyggie_execute_sql`, check whether the readonly role's grants got revoked accidentally.

### Sudden zero traffic to a tool

Not the same alert, but worth noting if you're here for tool-degraded reasons. Check `CYGGIE_MCP_ENABLED` — somebody may have disabled the entire MCP route.

## Mitigation while debugging

If a tool is so broken that it's actively hurting users, temporarily disable it:

- **Single tool:** there's no per-tool flag in V1. You'd need to comment out the `registerTool` call and redeploy. Heavy.
- **`cyggie_execute_sql`:** `fly secrets set CYGGIE_MCP_SQL_ENABLED=false -a cyggie-gateway`. Instant.
- **Entire MCP route:** `fly secrets set CYGGIE_MCP_ENABLED=false -a cyggie-gateway`. Heavier; affects every external MCP client.

Prefer fixing forward; flag-flipping is for genuine emergencies.

## Post-incident

- If a fix landed, drop a note in the team channel: which tool, what was broken, what changed.
- If patterns repeat (same tool keeps degrading), file an issue to add tool-specific monitoring / circuit-breaker.
- The plan flags `mcp_audit` retention as a V1 follow-up TODO. If you're noticing this runbook is hard to use because audit rows have aged out, that's the signal to land the retention policy.
